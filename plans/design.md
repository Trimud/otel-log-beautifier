## Problem Statement

Raw NDJSON logs from OpenTelemetry-instrumented services are hard to scan in the VS Code integrated terminal. Every existing VS Code log extension (Slog Viewer, LogLens, Smart-Log, Log File Highlighter, JSON Logs) works on files or webview panels, not the terminal. No extension exists that intercepts and reformats terminal output in-place.

## What Makes This Cool

Transparent, zero-config log beautification that just works. You open a terminal, run your service, and JSON log lines automatically become colorized, structured output with level highlighting, field indentation, and repeated-resource suppression. Non-JSON output passes through unchanged. No piping, no modified run commands, no context switching to a separate panel.

## Constraints

- Must use the Pseudoterminal API (only stable VS Code API for this)
- node-pty required for full PTY emulation (resize, signals, interactive programs)
- NDJSON only in v1 (one JSON object per line, no multi-line/pretty-printed JSON. Supports multiple logging framework formats as long as each log entry is a single line.)
- Parser/formatter must be pure functions (no VS Code API dependency, fully unit-testable)
- Platform-specific .vsix builds needed due to node-pty native compilation
- `\r\n` line endings required by Pseudoterminal

## Premises

1. The Pseudoterminal API is the only viable approach for in-terminal log beautification in VS Code (no simpler API exists). AGREED.
2. Automatic detection beats piping through CLI tools like pino-pretty, because you don't want to modify every run command. AGREED.
3. node-pty is worth the native compilation pain because you need real PTY emulation (resize, signals, interactive programs). AGREED.
4. For personal use, marketplace distribution can wait. Local .vsix install is fine for v1. AGREED.

## Cross-Model Perspective

An independent Claude subagent reviewed the design direction:

1. **Coolest unexplored version:** Live trace correlation in the terminal. Since OTel logs contain trace_id and span_id, clicking a trace ID could highlight every log line in that trace across scrollback. OSC hyperlinks could make trace IDs clickable links to Jaeger/Tempo. The terminal becomes a portal to your observability stack. (Deferred to future phases.)

2. **Key insight:** "Parser/formatter are pure functions, no VS Code API dependency" is the tell. The excitement is about clean architecture and composable data transformation, not the VS Code extension itself. The extension is the delivery vehicle; the engine is a standalone log-parsing pipeline.

3. **50% there:** node-pty + pino-pretty together cover half the work. node-pty gives the PTY wrapper. pino-pretty gives a battle-tested parser/formatter reference. The remaining 50%: VS Code Pseudoterminal glue, LineBuffer for PTY chunk reassembly, LogNormalizer for OTel/winston/bunyan, and JsonDetector for reliable JSON detection.

4. **Weekend build order:** PTY bridge first (the risk), then LineBuffer + JsonDetector, then minimal formatter, then integration. "The PTY bridge is the risk. Start with the hard part."

## Approaches Considered

### Approach A: Full Plan (CHOSEN)
All 5 implementation phases: scaffold and passthrough terminal, parser pipeline (LineBuffer, JsonDetector, LogNormalizer), formatter with ANSI colors, config/toggle/profile provider, and polish/packaging. Supports OTel + pino + winston + bunyan + generic JSON.
- Effort: L (human: ~1 week / CC: ~45 min)
- Risk: Low/Med
- Pros: Complete multi-format support, full UX, marketplace-ready
- Cons: More surface area before first value

### Approach B: Weekend MVP
OTel-only formatter with PTY bridge, line buffer, and JSON detector. Skip multi-format, config UI, packaging.
- Effort: S (human: ~2 days / CC: ~20 min)
- Risk: Low
- Pros: Fast to value, smaller surface area

### Approach C: Pipe Transform
CLI stdin/stdout tool (my-app | otel-pretty), no Pseudoterminal, no node-pty.
- Effort: S (human: ~1 day / CC: ~15 min)
- Risk: Low
- Cons: Requires modifying run commands, breaks premise #2

## Recommended Approach

**Approach A: Full Plan.** With CC, the effort delta between MVP and full plan is ~25 minutes. The user wants the complete thing. Build all 5 phases, all 5 log formats, full config/toggle/status bar UX.

### CommonLogRecord Interface

The normalized intermediate format used between parser and formatter:

```typescript
interface CommonLogRecord {
  readonly timestamp: string;        // ISO 8601 or epoch ms, always present
  readonly level: string;            // Normalized: TRACE | DEBUG | INFO | WARN | ERROR | FATAL
  readonly message: string;          // Log message body
  readonly traceId?: string;         // OTel trace ID (hex string)
  readonly spanId?: string;          // OTel span ID (hex string)
  readonly serviceName?: string;     // From resource.service.name or equivalent
  readonly resource?: Record<string, unknown>;   // OTel resource attributes
  readonly attributes?: Record<string, unknown>; // Remaining fields not mapped above
  readonly raw: Record<string, unknown>;         // Original parsed JSON for fallback
  readonly source: 'otel' | 'pino' | 'winston' | 'bunyan' | 'generic';
}
```

### Architecture

```
User input -> handleInput() -> node-pty child shell -> command executes
                                                          |
                                                    pty.onData(raw)
                                                          |
                                                    LineBuffer (assemble complete lines)
                                                          |
                                                    JsonDetector (is it JSON?)
                                                    |              |
                                                   NO             YES
                                                    |              |
                                              passthrough    LogNormalizer (OTel/pino/winston/bunyan)
                                                    |              |
                                                    |        LogFormatter (ANSI-colored output)
                                                    |              |
                                                    +------+-------+
                                                          |
                                                  onDidWrite -> VS Code terminal
```

### Implementation Phases

**Phase 1: Scaffold and Passthrough Terminal**
- Initialize project: package.json (engine ^1.85.0), tsconfig.json (ES2022, strict), esbuild config
- Add node-pty as dependency (mark external in esbuild)
- Implement ptyBridge.ts (wraps node-pty, exposes onData/write/resize/kill)
- Implement child_process.spawn fallback in ptyBridge.ts (used when node-pty fails to load). Fallback loses: terminal resize, signal forwarding, interactive programs (vim, less, top). Sufficient for the primary use case (running a service that outputs logs).
- PTY bridge error handling: child process exits with code 0 -> close terminal silently. Non-zero exit -> emit "[Process exited with code N]" via onDidWrite, then fire onDidClose. node-pty load failure -> fall back to child_process.spawn silently with a one-time warning in the output channel. Mid-session PTY error -> log to output channel, attempt graceful shutdown.
- Preserve VS Code shell integration sequences: OSC 633 (command detection: prompt start/end, command start/end, current directory). Pass these through unchanged so VS Code's built-in terminal features (command navigation, sticky scroll) continue working.
- Implement minimal beautifiedTerminal.ts (passthrough only, no formatting)
- Register otelLogBeautifier.openTerminal command
- Verify: F5 -> Command Palette -> "Open Beautified Terminal" -> working shell

**Phase 2: Parser Pipeline**
- lineBuffer.ts: accumulate PTY chunks, emit complete lines on \n (or \r\n). Flush behavior at 64KB: emit the accumulated partial content as a non-JSON passthrough line (prevents memory leaks from pathological input). Lines containing \n inside JSON string values are not a concern because NDJSON guarantees one object per physical line.
- jsonDetector.ts: fast-path reject (first char not { after stripping), strip ALL ANSI escape sequences (regex: /\x1b\[[0-9;]*[a-zA-Z]/g) from the line before detection, then try JSON.parse
- logNormalizer.ts: detect format by field signatures, normalize to CommonLogRecord
- Unit tests for all three modules

**Phase 3: Formatter**
- ansiColors.ts: level colors (ERROR=red, WARN=yellow, INFO=cyan, DEBUG=gray)
- logFormatter.ts: output layout with timestamp, level, message, indented fields
- resourceTracker.ts: compare the `resource` field of CommonLogRecord via string equality on JSON.stringify output (no cryptographic hash needed, this is an in-process comparison). When identical to previous line's resource, omit the resource fields entirely from output (no placeholder, just skip). When resource changes, print the full resource block. "Resource block" = `resource` field in CommonLogRecord (mapped from OTel `Resource`, pino doesn't have one, winston/bunyan: `hostname`+`name`+`pid` fields).
- fieldLayout.ts: configurable field order, value truncation (default maxValueLength: 120 chars), max nesting depth (default: 3). Default field order: [timestamp, level, serviceName, message, traceId, spanId, ...remaining attributes alphabetically].
- Wire formatter into beautifiedTerminal.ts pipeline
- Unit tests for formatter

**Phase 4: Config, Toggle, Profile**
- contributes.configuration in package.json: enabled, fieldOrder, hiddenFields, resourceDisplay, timestampFormat, maxValueLength, maxDepth, colors, shell, shellArgs
- settings.ts: typed config reader + onDidChangeConfiguration listener
- otelLogBeautifier.toggleBeautify command: flip formatted <-> raw passthrough
- Status bar item: $(sparkle) OTel: ON/OFF, visible when active terminal is ours
- TerminalProfileProvider: "OTel Beautified" in terminal dropdown

**Phase 5: Polish and Package**
- Edge cases: binary output detection (if a chunk contains NUL bytes (0x00) or bytes forming invalid UTF-8 sequences, bypass line buffer and passthrough raw to avoid corrupting binary data from commands like `cat binary_file`), long lines (>64KB passthrough without JSON detection), 16ms output batching (debounce between formatter output and onDidWrite. Formatted lines are accumulated in a buffer and flushed together as a single onDidWrite call every 16ms, approximately 1 frame at 60fps. The batching sits AFTER the parse/format pipeline, not before LineBuffer. Each PTY onData event is processed through LineBuffer/JsonDetector/formatter immediately, but the rendered output is held until the next 16ms tick.)
- Fallback already implemented in Phase 1 (child_process.spawn when node-pty unavailable)
- Platform-specific .vsix via vsce package --target: win32-x64, win32-arm64, linux-x64, linux-arm64, darwin-x64, darwin-arm64
- README with screenshots and config docs

### Log Format Detection

| Format  | Detected by                               | Timestamp            | Level                  | Message            |
|---------|------------------------------------------|----------------------|------------------------|--------------------|
| OTel    | severity_text or severity_number          | timestamp            | severity_text/level    | message/body       |
| Pino    | numeric level (10-60) + pid               | time (epoch ms)      | level (numeric)        | msg                |
| Winston | string level + message                    | timestamp            | level                  | message            |
| Bunyan  | numeric level + hostname + name           | time (ISO)           | level (numeric)        | msg                |
| Generic | fallback                                  | timestamp/time/ts/@timestamp | level/lvl/severity | message/msg/text |

### Risks and Mitigations

| Risk                           | Mitigation                                                  |
|--------------------------------|-------------------------------------------------------------|
| node-pty native compilation    | Platform-specific .vsix + child_process.spawn fallback      |
| High-volume log performance    | Fast-path JSON rejection, 16ms output batching              |
| Shell integration breaks       | Pass through shell integration escape sequences unchanged   |
| ANSI codes confuse detection   | Strip ANSI before detection, preserve for non-JSON passthrough |

## Success Criteria

1. Open beautified terminal, run app outputting JSON logs -> colorized and formatted
2. Normal shell commands (ls, git) -> output unchanged
3. Toggle off -> raw JSON -> toggle on -> formatted again
4. Error log -> red with expanded details
5. Repeated resource block -> shown once then suppressed
6. Change hiddenFields in settings -> fields disappear
7. npm test -> all parser/formatter unit tests pass
8. Performance: handle 10,000 lines/second without visible lag or dropped output (test with a loop printing JSON lines)

## Accepted Scope Expansions (from CEO review cherry-picks)

1. **Log level filter** - Command palette commands: "OTel: Show Only ERROR" / "Show Only WARN+" / "Show All Levels". Conditional in the formatter pipeline that suppresses lines below the threshold. Add to Phase 4 (Config, Toggle, Profile).

2. **Extract core as npm package** - Move parser/ and formatter/ into a `packages/core` workspace package (`@otel-log-beautifier/core`). VS Code extension imports from the package. Enables CLI, CI, and web consumers. Restructure as monorepo in Phase 1.

3. **Clickable file paths in stack traces** - Register a TerminalLinkProvider that detects file:line patterns in beautified output and opens them in the editor. Ensures we don't regress VS Code's built-in link detection. Add to Phase 3 (Formatter).

## Additions from Outside Voice Review (CEO review cross-model)

4. **Alternate screen buffer detection (CRITICAL)** - Detect ESC[?1049h (enter alternate screen) and ESC[?1049l (leave alternate screen). When alternate screen is active, bypass the ENTIRE parse pipeline (LineBuffer, JsonDetector, formatter) and passthrough raw PTY data directly. Also detect raw mode. This prevents corruption of interactive programs (vim, less, top, SSH). Add to Phase 1 (PTY bridge).

5. **Worker thread for parse/format pipeline** - Move the parse/format pipeline (LineBuffer, JsonDetector, LogNormalizer, LogFormatter) to a Node.js Worker thread to avoid blocking the VS Code extension host at sustained throughput. The main thread handles PTY I/O and onDidWrite; the worker handles CPU-intensive parsing. Add to Phase 2 (Parser Pipeline).

6. **Sort keys before resource comparison** - Use `JSON.stringify(resource, Object.keys(resource).sort())` instead of plain `JSON.stringify` to handle different key insertion orders across code paths.

7. **Debug mode** - Config flag `otelLogBeautifier.debug: true` that logs classification decisions to a VS Code Output Channel for troubleshooting.

8. **Filter status line** - When level filter suppresses all output, show periodic dim status: "[Filtering: ERROR only. N lines suppressed since last error.]"

9. **Extension deactivation handling** - On deactivate(), gracefully kill all node-pty child processes and show "[Extension deactivated]" in open beautified terminals.
